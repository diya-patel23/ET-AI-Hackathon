# Unified Asset & Operations Brain

Industrial Knowledge Intelligence Platform — document ingestion, RAG copilot,
knowledge graph, and three reasoning agents (Root Cause Analysis, Maintenance
Intelligence, Compliance) over industrial maintenance/inspection/safety
documents.

This is the **Tier 1 + Tier 2** scope from the architecture doc: everything
below is real, working code — not a mockup. It was built and smoke-tested
end-to-end (ingestion → knowledge graph → search → dashboard → all three
agents) before being handed to you. See the bottom of this file for exactly
what was verified and what you should verify yourself.

```
industrial-knowledge-platform/
├── backend/     FastAPI + SQLite + Chroma (Python)
├── frontend/    Next.js 16 + Tailwind + React Flow + Recharts (TypeScript)
└── README.md    this file
```

---

## 1. Prerequisites

- Python 3.11+
- Node.js 18+
- `tesseract-ocr` installed system-wide, only if you want OCR on scanned
  images: `apt-get install tesseract-ocr` (Ubuntu/Debian) or `brew install
  tesseract` (macOS). Everything else works fine without it — OCR just
  becomes a no-op for image uploads.

## 2. Backend setup

```bash
cd backend
python3 -m venv venv
source venv/bin/activate          # Windows: venv\Scripts\activate
pip install -r requirements.txt

cp .env.example .env
# Optionally edit .env and add LLAMA_API_KEY — see section 4.

# Generates ~25 synthetic documents (maintenance logs, inspection reports,
# safety/OEM manual excerpts) around a recurring set of 8 equipment items,
# and ingests them through the REAL pipeline (parse, chunk, embed, extract
# entities, build the graph). Takes under a minute.
python -m seed.run_seed

# Start the API
uvicorn app.main:app --reload --port 8000
```

Check it's alive: `curl http://localhost:8000/api/health` should return
`{"status":"ok", "llm_configured": false, "embedding_backend": "..."}`.

Interactive API docs (Swagger UI, auto-generated by FastAPI): open
`http://localhost:8000/docs` in a browser. Every endpoint listed in the
architecture doc is there and callable directly — useful for demoing the
API layer on its own, or for debugging without the frontend.

## 3. Frontend setup

In a second terminal:

```bash
cd frontend
npm install
# .env.local is already checked in, pointing at http://localhost:8000.
# Edit it if your backend runs elsewhere.
npm run dev
```

Open `http://localhost:3000`. You should see real data immediately (26
documents, 8 equipment items, a failure trend chart) because you seeded the
backend in step 2.

Pages: `/` dashboard, `/copilot` RAG chat, `/documents` upload + browse,
`/graph` knowledge graph explorer (search "Pump P204"), `/agents` Root
Cause / Maintenance / Compliance.

## 4. Turning on the Local LLM

The application uses **Ollama** to run the language model locally instead of relying on a hosted API or API key. This allows the LLM layer to work without sending industrial documents or data to an external service.

Without Ollama running, the app is still partially functional. Document ingestion, search, the knowledge graph, causal-chain structure, risk scores, and compliance pass/fail checks continue to work. However, copilot answers and agent narratives will show a `[LLM not available]` placeholder instead of a generated response.

To turn on the local LLM:

1. Install Ollama from the official website.
2. Pull the required model:

```bash
ollama pull llama3.2
```

3. Start Ollama if it is not already running:

```bash
ollama serve
```

4. Make sure the backend is configured to use the local Ollama model:

```env
OLLAMA_MODEL=llama3.2
OLLAMA_BASE_URL=http://localhost:11434
```

The backend communicates with Ollama through its local HTTP API. No API key is required.

After starting Ollama, restart the backend.

if you want the LLM entity-enrichment pass to be applied retroactively to the seeded documents as well.

## 5. Uploading your own documents

Drop real PDFs/DOCX/XLSX/CSV/images into `/documents` in the UI, or `POST
/api/ingest` directly. For the knowledge graph and agents to recognize
equipment automatically, your documents should follow the same light
conventions the seed generator uses — these aren't hard requirements, just
what the current regex extractor looks for (see
`backend/app/ingestion/entity_extractor.py` to extend the vocabulary):

- Equipment mentioned as `<Type> <Code>`, e.g. `Pump P204`, `Compressor C305`
- `Engineer: <name>` or `Inspected by: <name>`
- `Plant <n>`
- `Failure Code: FC-1234` or bare `FC-1234`
- Standards as `OSHA 1910.132`, `ISO 55001`, `API 610`

Anything that doesn't match these patterns still gets ingested, chunked, and
is fully searchable via RAG — it just won't automatically appear as a node
in the knowledge graph.
