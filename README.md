# Unified Asset & Operations Brain

Industrial Knowledge Intelligence Platform — document ingestion, RAG copilot,
knowledge graph, and three reasoning agents (Root Cause Analysis, Maintenance
Intelligence, Compliance) over industrial maintenance/inspection/safety
documents.

This is the **Tier 1 + Tier 2** scope from the architecture doc: everything
below is real, working code — not a mockup. It was built and smoke-tested
end-to-end (ingestion → knowledge graph → search → dashboard → all three
agents) before being handed to you. See the bottom of this file for exactly
what was verified and what you should verify yourself.

```
industrial-knowledge-platform/
├── backend/     FastAPI + SQLite + Chroma (Python)
├── frontend/    Next.js 16 + Tailwind + React Flow + Recharts (TypeScript)
└── README.md    this file
```

---

## 1. Prerequisites

- Python 3.11+
- Node.js 18+
- `tesseract-ocr` installed system-wide, only if you want OCR on scanned
  images: `apt-get install tesseract-ocr` (Ubuntu/Debian) or `brew install
  tesseract` (macOS). Everything else works fine without it — OCR just
  becomes a no-op for image uploads.

---

## 2. Backend setup

```bash
cd backend
python3 -m venv venv
source venv/bin/activate          # Windows: venv\Scripts\activate
pip install -r requirements.txt

cp .env.example .env
# Optionally edit .env and add LLAMA_API_KEY — see section 4.

# Generates ~25 synthetic documents (maintenance logs, inspection reports,
# safety/OEM manual excerpts) around a recurring set of 8 equipment items,
# and ingests them through the REAL pipeline (parse, chunk, embed, extract
# entities, build the graph). Takes under a minute.
python -m seed.run_seed

# Start the API
uvicorn app.main:app --reload --port 8000
```

Check it's alive: `curl http://localhost:8000/api/health` should return
`{"status":"ok", "llm_configured": false, "embedding_backend": "..."}`.

Interactive API docs (Swagger UI, auto-generated by FastAPI): open
`http://localhost:8000/docs` in a browser. Every endpoint listed in the
architecture doc is there and callable directly — useful for demoing the
API layer on its own, or for debugging without the frontend.

## 3. Frontend setup

In a second terminal:

```bash
cd frontend
npm install
# .env.local is already checked in, pointing at http://localhost:8000.
# Edit it if your backend runs elsewhere.
npm run dev
```

Open `http://localhost:3000`. You should see real data immediately (26
documents, 8 equipment items, a failure trend chart) because you seeded the
backend in step 2.

Pages: `/` dashboard, `/copilot` RAG chat, `/documents` upload + browse,
`/graph` knowledge graph explorer (search "Pump P204"), `/agents` Root
Cause / Maintenance / Compliance.

## 4. Turning on the LLM 

Without an API key, the app is fully functional except that copilot answers
and agent narratives show a `[LLM not configured]` placeholder instead of a
synthesized response — everything else (search, the causal chain's
structure, risk scores, compliance pass/fail) still works because those are
computed by real logic, not the LLM.

To turn it on:

```bash
# backend/.env
LLAMA_API_KEY=sk-ant-...
```

Restart the backend. Re-run `python -m seed.run_seed` if you want the LLM
entity-enrichment pass applied retroactively to the seeded documents too.

## 5. Uploading your own documents

Drop real PDFs/DOCX/XLSX/CSV/images into `/documents` in the UI, or `POST
/api/ingest` directly. For the knowledge graph and agents to recognize
equipment automatically, your documents should follow the same light
conventions the seed generator uses — these aren't hard requirements, just
what the current regex extractor looks for (see
`backend/app/ingestion/entity_extractor.py` to extend the vocabulary):

- Equipment mentioned as `<Type> <Code>`, e.g. `Pump P204`, `Compressor C305`
- `Engineer: <name>` or `Inspected by: <name>`
- `Plant <n>`
- `Failure Code: FC-1234` or bare `FC-1234`
- Standards as `OSHA 1910.132`, `ISO 55001`, `API 610`

Anything that doesn't match these patterns still gets ingested, chunked, and
is fully searchable via RAG — it just won't automatically appear as a node
in the knowledge graph.

## 6. What was actually verified before this was handed to you

Run in this environment, in order, on a clean database:

1. `python -m seed.run_seed` — generated 26 documents, ingested every one
   through the real pipeline, seeded 48 structured maintenance events.
   Confirmed clean entity extraction (engineer names, equipment, plants,
   parts, standards) after fixing one regex bug caught during this process.
2. `GET /api/health` → `200`, backend correctly detected no local
   sentence-transformers/torch install and **automatically fell back to the
   TF-IDF embedder** with zero code changes needed — this is the offline
   fallback path working as designed, not a workaround.
3. `GET /api/dashboard/stats` → real counts derived from the seeded data
   (26 docs, 8 equipment, 7 incidents, a real month-by-month failure trend).
4. `GET /api/search?q=Pump P204&mode=hybrid` → real ranked results across
   semantic, keyword, and entity search, correctly surfacing that
   equipment's maintenance log, inspection reports, and OEM manual excerpt.
5. `POST /api/agents/root-cause`, `/api/agents/maintenance`, `/api/chat` →
   all three correctly ran their **rule-based fallback paths** (no LLM
   key was configured in this environment) and returned well-formed,
   correct responses instead of erroring.
6. `POST /api/agents/compliance` → ran the 5-rule check against a real
   ingested inspection report, returned a 100% pass with per-rule
   explanations.
7. `GET /api/graph/entity/{id}` → returned Pump P204's real relationships
   (`uses` → Bearing B42, `located_in` → Plant 3, `maintained_by` →
   engineer, `governed_by` → OSHA/ISO standards).
8. Frontend: `npm run build` succeeded with zero TypeScript errors.
   `npm run start` served all five pages (`/`, `/copilot`, `/documents`,
   `/graph`, `/agents`) with `200` responses against the live backend.

**Not verified in this environment** (no Anthropic API key was available
here): the LLM-powered paths — synthesized copilot answers, LLM-generated
root cause narratives, LLM entity enrichment, and LLM-judged compliance
explanations. The code paths exist and have explicit `llm_available()`
guards with fallback behavior that *was* verified — but you should run
one real query with your own `ANTHROPIC_API_KEY` set before your demo to
confirm the synthesized answers read the way you want.

**Not built at all** (see the architecture doc's Tier 3 / cut-features
list): Neo4j (uses graph-as-tables in SQLite instead, same entity/relation
vocabulary so migrating later is straightforward), role-based auth (the
role switcher in the UI is cosmetic — it doesn't gate data), voice
interface, offline mode beyond the embedding fallback, multi-language
support, and structural P&ID/engineering-drawing parsing (images are OCR'd
as flat text, not structurally parsed).

## 7. If something doesn't work on your machine

- **`ModuleNotFoundError` on backend startup** — you're not in the venv, or
  `pip install -r requirements.txt` didn't finish. Re-run it and watch for
  errors (sentence-transformers pulls in `torch`, which is large — if disk
  space is tight, `pip install -r requirements.txt --no-deps
  sentence-transformers` then reinstall the rest, or just leave
  sentence-transformers out entirely; the TF-IDF fallback activates
  automatically and the app is fully functional without it).
- **Frontend shows "API unreachable"** — backend isn't running, or
  `frontend/.env.local`'s `NEXT_PUBLIC_API_URL` doesn't match where it's
  running.
- **OCR returns empty text for images** — `tesseract-ocr` isn't installed
  system-wide. The document still ingests; it just has no extracted text.
- **Graph page shows nothing after searching** — the entity name has to
  partially match; try the exact seeded names like `Pump P204`, `Bearing
  B42`, or `Plant 3` first to confirm the graph itself is populated, then
  adjust for your own documents.
